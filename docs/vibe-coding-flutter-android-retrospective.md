# 从 lzhᴗh 开始学习移动端开发

## 项目概况

`lzhᴗh` 是一个轻量、自用、离线优先的 Flutter Android 文件转换工具。它没有追求复杂架构和大量功能，而是围绕真实手机使用场景逐步完成：

- JPG、PNG、WebP 图片格式转换
- 图片压缩
- 多张图片生成 PDF
- 视频提取为 MP3、M4A、WAV
- 轻量 JSON 转换记录
- 结果文件打开、发送原文件和删除
- 长任务进度、取消和异常提示
- 大文件、像素数量、磁盘空间和内存保护
- ARM32、ARM64 分包构建

项目始终坚持以下约束：

- 优先真机调试，不安装模拟器
- Gradle 内存上限 3 GB，最多 2 个 worker
- 不升级到 AGP 9
- 优先使用 Android 原生能力，谨慎增加 Flutter 插件
- 数据结构和架构保持简单，适合个人维护

## 开发过程

### 第一阶段：建立最小可运行流程

最初先完成界面、文件选择和页面跳转，验证最基本的用户流程：

`选择文件 -> 设置参数 -> 开始处理 -> 查看结果`

这一阶段最重要的不是界面是否精致，而是尽早确认 Flutter 工程、Android 构建环境和真机安装链路可以工作。

### 第二阶段：实现真实图片处理

图片功能通过 Flutter `MethodChannel` 调用 Kotlin 原生代码完成。这样避免引入较大的图片处理插件，也更容易控制内存。

实现时加入了：

- 单工作线程处理，防止并行解码占满内存
- 图片尺寸和像素数量限制
- EXIF 方向处理
- 输出质量和尺寸参数
- 临时文件失败清理
- 明确的异常信息

这一步让我认识到，图片转换不是简单修改扩展名。真正的转换必须解码源文件，再以目标编码重新写出，并通过文件头或其他工具确认实际格式。

### 第三阶段：图片转 PDF 与转换记录

多图片生成 PDF 时同样需要控制内存，不能一次把所有原图完整解码并长期保存在内存中。

转换记录没有引入数据库，而是使用 JSON 文件：

- 最多保留 200 条记录
- 写入临时文件后再替换正式文件
- 单条记录损坏时尽量跳过
- 整体 JSON 损坏时备份旧文件
- 删除结果文件时同步更新记录状态

对于个人工具，这种实现比数据库更轻，也更容易检查和迁移。

### 第四阶段：打开、分享和删除闭环

结果文件生成后，还必须能真正被用户使用。项目完善了：

- 使用 `FileProvider` 把私有文件临时授权给其他应用
- 使用只读 `content://` URI 打开文件
- 通过 Android 分享面板发送原文件
- 删除结果文件并同步历史记录
- 文件不存在时显示可理解的提示

真机测试曾出现：

`Failed to find configured root that contains ...`

原因是 `FileProvider` 配置的根目录没有覆盖 Flutter 实际使用的 `app_flutter` 路径。修正 XML 根路径后，打开和分享功能才真正闭环。

### 第五阶段：视频提取音频

视频处理使用精简的 FFmpeg 音频组件，并重点检查许可证、APK 体积和内存使用。

实现策略包括：

- 只读取第一个音频流，不解码视频画面
- MP3、M4A、WAV 分别使用适当参数
- FFmpeg 限制为单线程
- 提供执行进度、取消和异常提示
- 取消或失败时删除不完整输出
- 限制超大文件和超长媒体
- 转换前估算输出大小和剩余磁盘空间
- 按 ABI 构建，避免通用 APK 包含所有架构

这一阶段最大的认识是：第三方库不能只看“能不能运行”，还要看：

- 制品是否长期可下载
- LGPL、GPL 等许可证是否符合分发方式
- 编解码器是否真的包含在构建中
- 是否支持目标 Android ABI
- 是否显著增大 APK
- 是否适配 Android 新的页面大小等平台要求

### 第六阶段：体验和发布完善

最后补充了：

- 首页状态和历史筛选
- 长任务进度展示
- 权限、空路径、文件损坏等边界处理
- 应用名称、图标和版本信息
- Debug 与 Release 构建检查
- ARM64 真机安装包

项目最终通过了 `flutter analyze`、自动测试和 Release APK 构建。

## Vibe Coding 中遇到的主要问题

### 1. 功能实现不等于用户目标实现

应用确实生成了 PNG 文件，但用户把文件发送到微信，再从微信保存到相册后，看到的格式可能又发生变化。

这里至少存在三段行为：

1. 应用生成什么文件
2. Android 分享系统传递什么文件
3. 微信接收、预览和保存时是否重新编码

第三方应用可能压缩或重编码媒体。应用只能保证自己生成和发送的原文件格式，不能保证另一个应用保存后的副本完全不变。

因此验收不能只检查“代码执行成功”，而要检查用户完整操作路径，并明确哪一步属于本应用控制范围。

### 2. AI 很容易快速实现错误理解

“相册里没有变化”可能有多种含义：

- 没有保存到相册
- 相册没有刷新
- 图片内容看起来相同
- 文件扩展名没有变化
- 第三方应用保存时改了格式

如果没有先复述用户的具体操作步骤，AI 可能很快实现一个并非用户需要的“自动导出到某个相册目录”功能。

更可靠的方式是先确认：

- 用户点了哪些按钮
- 文件经过了哪些应用
- 最终在哪里观察结果
- 成功的可观察标准是什么

### 3. Android 文件系统比桌面端复杂

项目中同时接触了：

- 应用私有目录
- Flutter `app_flutter` 目录
- `FileProvider`
- `content://` URI
- Android `MediaStore`
- 系统文件选择器 SAF
- 第三方应用创建的副本

这些位置和机制不能互相替代。私有目录适合应用管理文件，但相册通常看不到；`FileProvider` 适合临时分享，但不会自动把文件加入媒体库；`MediaStore` 适合公开媒体；SAF 适合让用户自己选择保存位置。

### 4. 依赖管理不只是添加一行配置

FFmpeg 组件曾遇到下载、仓库可用性和本地 AAR 接入问题。实际开发还需要处理：

- Maven 下载失败
- 本地缓存位置和磁盘空间
- 手动下载制品
- Java 依赖缺失
- 许可证核对
- APK 体积变化
- ABI 拆分

Vibe Coding 容易让人把“依赖已写入 Gradle”误认为工作完成，但真正完成必须包括构建、打包和真机运行。

### 5. Debug 包不能代表最终体积

包含调试信息和多 ABI 的通用 Debug APK 会非常大。Release 分包后，ARM64 安装包才更接近用户实际拿到的体积。

所以讨论 APK 大小时必须说明：

- Debug 还是 Release
- 通用包还是单 ABI 包
- 是否包含符号和调试资源

### 6. 构建环境也是软件的一部分

开发过程中遇到的环境问题包括：

- SDK、Gradle 和 Maven 缓存权限
- 下载失败或网络不稳定
- PowerShell 执行限制
- Windows 路径问题
- Python 启动器不可用
- 缓存占用系统盘

将下载和缓存统一放在 D 盘，并保留手动下载 AAR/JAR 的后备路径，使构建过程更可控。环境配置应该被记录，而不是只存在于某一次聊天里。

## 这次做对的事情

- 先阅读现有代码，再沿用项目已有模式
- 以小阶段完成真实可运行的纵向功能
- 优先使用 Kotlin 和 Android 原生 API
- 使用 JSON 而不是为少量历史记录引入数据库
- 为 FFmpeg 设置单线程、大小限制和取消清理
- 每个阶段运行格式化、静态分析、测试和构建
- 根据真机截图定位 `FileProvider` 配置错误
- 区分应用生成的原文件与第三方应用重新保存的副本
- 不把未重新构建、未验证的旧 APK 当作最新成果

## 可以改进的地方

- 在写代码前先为每个功能写出可观察的验收标准
- 图片格式转换增加文件头校验测试，而不只检查扩展名
- 更早区分 Debug、Release、通用包和分 ABI 包
- 增加 Android instrumentation 测试，覆盖 `FileProvider` 和 Intent
- 把私有保存、公开导出、分享给第三方设计成三个独立场景
- 及时清理 README 中已经过时的计划和依赖说明
- 遇到含义不明确的反馈时，先还原完整用户操作路径

## 建议的移动端学习顺序

### 1. Flutter 基础

- Widget、布局、主题和导航
- `StatefulWidget` 与简单状态管理
- `Future`、`Stream` 和生命周期
- 文件选择、异步任务和错误提示
- Dart 单元测试与 Flutter Widget 测试

### 2. Android 基础

- Activity、Manifest 和权限
- 应用私有存储与共享存储
- Intent、URI 和 `FileProvider`
- `MediaStore` 与 SAF
- Gradle、签名、版本号和 ABI

### 3. Flutter 与原生协作

- `MethodChannel`
- `EventChannel`
- Kotlin 后台线程
- 进度事件、取消和异常映射
- Flutter 页面销毁后的资源释放

### 4. 发布质量

- 真机兼容性测试
- 大文件和低内存测试
- 权限与隐私检查
- APK 体积和性能分析
- 第三方许可证
- Release 签名和版本管理

## 后续练习建议

- 使用 `MediaStore` 实现明确的“导出到相册”
- 使用 SAF 实现“另存到用户选择的位置”
- 为 `FileProvider` 编写 Android 集成测试
- 准备 JPG、PNG、WebP 样本并校验真实文件头
- 配置正式 Release keystore
- 学习使用 `adb logcat` 定位真机异常

## 核心结论

Vibe Coding 不是用一句提示词直接得到成品，而是一个循环：

`澄清目标 -> 小步实现 -> 检查产物 -> 真机验证 -> 修正理解 -> 沉淀经验`

AI 能显著加快实现速度，也会同样放大错误理解。对移动端开发来说，最重要的能力不是记住所有 API，而是能够分清 Flutter、Android 系统、第三方应用和构建环境各自负责什么，并为真实用户流程建立可验证的标准。
